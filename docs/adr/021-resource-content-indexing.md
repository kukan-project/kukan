# ADR-021: リソースデータ本体の全文検索インデックス

## ステータス

**承認済み（Accepted）** — 2026-04-18

## コンテキスト

KUKAN の検索は現在メタデータのみ（データセット名・説明・タグ・リソース名等）を対象としている。
CSV の中身やテキストファイルの内容でデータセットを検索したいというニーズがある。

Pipeline の Index ステップ（現在 no-op）を復活させ、抽出したテキストを OpenSearch に投入する設計を行った。

### 検討した判断ポイント

7 つの主要な設計判断がある。

---

## 判断 1: 既存インデックス拡張 vs 別インデックス

### 選択肢

| 方式 | 構成                                                                   | メリット                                             | デメリット                                                               |
| ---- | ---------------------------------------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------ |
| A    | `kukan-packages` の nested resources に `extractedText` フィールド追加 | クエリが単一インデックスで完結                       | ドキュメントサイズ肥大化（100KB × リソース数）、メタデータ検索の性能劣化 |
| B    | **別インデックス `kukan-resources` を新設**                            | メタデータ検索に影響なし、独立したライフサイクル管理 | 2 インデックスの `msearch` + アプリ層マージが必要                        |

### 決定: 方式 B — 別インデックス `kukan-resources`

コンテンツテキストは 1 リソースあたり最大 100KB。データセットに 10 リソースあれば 1MB 近くになり、
`kukan-packages` のドキュメントサイズが桁違いに増大する。メタデータ検索（大多数のクエリ）の性能を維持するため、
コンテンツは専用インデックスに分離する。

`msearch` API で 1 ラウンドトリップに 2 クエリを乗せるため、レイテンシ増加は最小限。

---

## 判断 2: コンテンツテキストの永続化先

### 選択肢

| 方式 | 構成                                                | メリット                                                               | デメリット                                                       |
| ---- | --------------------------------------------------- | ---------------------------------------------------------------------- | ---------------------------------------------------------------- |
| A    | DB に `resource_content` テーブルを作成             | 再インデックスが高速（SELECT のみ）、PostgreSQL 全文検索の対象にできる | DB 容量が増大（テキスト + GIN インデックスで元サイズの 5〜7 倍） |
| B    | **DB に保持しない。Pipeline → OpenSearch 直接投入** | DB 容量増加なし、実装がシンプル                                        | 再インデックス時に S3 から再ダウンロード・再パースが必要         |

### 決定: 方式 B — DB にコンテンツテキストを保持しない

再インデックスの実行頻度は極めて低い（OpenSearch 障害やマッピング変更時のみ）。
S3 からの再ダウンロードコストは低く、パイプラインの Index ステップと同じロジックを再利用できる。
DB 容量の節約と実装のシンプルさを優先する。

切り詰め情報（インデックス済みか、元サイズ、切り詰めサイズ）は既存の `resource_pipeline.metadata` JSONB に記録する。

---

## 判断 3: PostgreSQL フォールバックでのコンテンツ検索

### 選択肢

| 方式 | 構成                                                                       | メリット                                | デメリット                                                                |
| ---- | -------------------------------------------------------------------------- | --------------------------------------- | ------------------------------------------------------------------------- |
| A    | `resource_content` テーブル + `gin_trgm_ops` GIN インデックスで ILIKE 検索 | OpenSearch なしでもコンテンツ検索可能   | DB 容量肥大（100KB × N リソース × GIN 3〜5 倍）、1 万リソース超で性能問題 |
| B    | **PostgreSQL ではコンテンツ検索しない**                                    | DB 容量増加なし、`postgres.ts` 変更不要 | 閉域網ユーザーはコンテンツ検索を利用できない                              |

### 決定: 方式 B — PostgreSQL ではコンテンツ検索しない

PostgreSQL モードは主に小規模デプロイ（開発環境、閉域網 PoC）向け。
メタデータ検索で実用上十分であり、コンテンツ検索は OpenSearch の付加価値として位置づける。

pg_trgm の GIN インデックスは 100KB テキスト × 数千リソースで数百 MB〜数 GB に達し、
小規模環境での DB サイズを不必要に膨らませる。

---

## 判断 4: コンテンツテキストのサイズ上限

### 検討

OpenSearch のドキュメントサイズに関する実用上の制約:

| サイズ  | 評価                                       |
| ------- | ------------------------------------------ |
| ~10 KB  | 理想的。一般的な検索ドキュメント           |
| ~100 KB | 良好。長文記事レベル                       |
| ~500 KB | 許容範囲だがインデックス速度・メモリに影響 |
| ~1 MB   | 上限付近。GC プレッシャー増大              |

日本語テキスト 100KB ≈ 約 3〜5 万文字 ≈ A4 で 40〜60 ページ分。
CSV なら数千行のヘッダー + データを十分カバーできる。

### 決定: 100KB を上限とする

- `MAX_CONTENT_INDEX_SIZE = 100 * 1024`（100KB）
- 100KB を超えるファイルは先頭 100KB で切り詰め
- 切り詰め情報を `resource_pipeline.metadata` に記録:
  - `contentIndexed`: boolean — インデックスされたか
  - `contentType`: string — `'tabular' | 'text' | 'manifest'`
  - `contentOriginalSize`: number — 元テキストのバイト数
  - `contentIndexedSize`: number — 実際にインデックスしたバイト数
  - `contentTruncated`: boolean — 切り詰められたか

---

## 対象フォーマット

| フォーマット   | 抽出方法                         | contentType |
| -------------- | -------------------------------- | ----------- |
| CSV / TSV      | UTF-8 変換 → ヘッダー + 行データ | `tabular`   |
| TXT / MD       | UTF-8 変換 → プレーンテキスト    | `text`      |
| HTML / HTM     | UTF-8 変換 → タグ除去            | `text`      |
| JSON / GeoJSON | そのまま                         | `text`      |
| XML            | UTF-8 変換                       | `text`      |
| ZIP            | マニフェストのファイル名一覧     | `manifest`  |
| PDF / Office   | 対象外（将来対応）               | —           |

## 判断 5: リソースメタデータの格納先

### 選択肢

| 方式 | 構成                                                                                            | メリット                                                               | デメリット                                                                             |
| ---- | ----------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| A    | リソースメタデータは `kukan-packages` の nested に残し、コンテンツのみ `kukan-resources` に格納 | 既存の `kukan-packages` 構造を変更しない                               | リソース検索ロジックが 2 箇所に分散（メタデータは nested、コンテンツは別インデックス） |
| B    | **リソースメタデータもコンテンツも `kukan-resources` に統合**                                   | 「リソースの検索は `kukan-resources`」という単一責務。マッピング簡素化 | `kukan-packages` から nested resources を廃止する構造変更が必要                        |

### 決定: 方式 B — リソースメタデータを `kukan-resources` に統合

リソースの name/description 検索とコンテンツ検索を同じインデックスで行うことで：

- 検索ロジックが一元化される（nested クエリ廃止）
- スコアリングが統一される（メタデータとコンテンツのブーストを同じ `multi_match` で制御）
- データセット CUD 時に全リソースを再投入する必要がなくなる
- 将来のリソース単体検索 API にも対応しやすい

`kukan-packages` からは nested `resources` フィールドを廃止し、`formats` 配列のみ維持する（ファセット用）。

### スコアリング設計

2 インデックスのスコアは独立して計算されるため、アプリ層でマージ時に統合する。

**リソース側のフィールドブースト:**

```
name^3 > description^2 > extractedText（ブーストなし）
```

**データセットの最終スコア:**

```
finalScore = packages_score + max(resource_scores) * RESOURCE_BOOST
```

- `packages_score`: `kukan-packages` の BM25 スコア
- `max(resource_scores)`: そのデータセットに属するリソースの最高スコア
- `RESOURCE_BOOST`: 0.3〜0.5（設定可能な定数、運用しながら調整）

これにより、メタデータで完全マッチするデータセットがコンテンツにたまたま含まれるだけのデータセットより上位に来る。

## 判断 6: 3 インデックス分離（コンテンツの独立）

### 背景

初期実装では `kukan-resources` にリソースメタデータとコンテンツテキスト（`extractedText`）を
同居させていたが、運用上の問題が判明した。

**問題**: メタデータ再構築（`POST /admin/reindex`）時に `kukan-resources` を全削除→再投入すると、
コンテンツテキスト（Pipeline で生成、S3 から再取得が必要）も消失する。
コンテンツ再処理なしでメタデータだけ再構築したいケースに対応できない。

### 選択肢

| 方式 | 構成                                                           | メリット                                                   | デメリット                                             |
| ---- | -------------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------ |
| A    | upsert で上書き（全削除しない）                                | コンテンツが保持される                                     | 孤立ドキュメントが残る可能性。partial update が複雑    |
| B    | **コンテンツを `kukan-contents` に分離（3 インデックス構成）** | メタデータ再構築がコンテンツに影響しない。関心の分離が明確 | msearch が 3 クエリに増加（ただし 1 ラウンドトリップ） |

### 決定: 方式 B — 3 インデックス構成

```
kukan-packages  → データセットメタデータ（title, notes, tags, org, ...）
kukan-resources → リソースメタデータ（name, description, format）
kukan-contents  → 抽出テキスト（extractedText, contentType）
```

各インデックスのライフサイクルが独立する:

| 操作             | packages      | resources     | contents                 |
| ---------------- | ------------- | ------------- | ------------------------ |
| メタデータ再構築 | 全削除→再投入 | 全削除→再投入 | **触らない**             |
| コンテンツ再処理 | —             | —             | 全削除→Pipeline で再投入 |
| 通常の CUD       | upsert        | upsert        | —                        |
| Pipeline 完了    | —             | —             | upsert                   |

将来の拡張（チャンク分割、Embedding）においても、`kukan-contents` の内部構造を変えるだけで
`kukan-packages` / `kukan-resources` は影響を受けない。

## 判断 7: 将来のチャンク分割と Embedding 対応方針

### 背景

Phase 5（AI & セマンティック検索）では、Embedding ベクトルによる意味検索を計画している。
また、PDF 等の大きなドキュメントのコンテンツ検索にも対応したい。

### チャンク分割の必要性

**キーワード検索（BM25）** と **Embedding（kNN）** では最適なチャンクサイズが異なる。

| 用途                   | 最適サイズ             | 理由                                                                     |
| ---------------------- | ---------------------- | ------------------------------------------------------------------------ |
| BM25 キーワード検索    | ~100 KB                | 長いテキストの方がマッチ機会が多い                                       |
| Embedding ベクトル検索 | ~500 tokens（~1.5 KB） | モデルの入力上限（Titan v2: 8,192 tokens）。大きすぎると意味が希釈される |

1 ドキュメントに全文格納する方式（方式 B）は、highlight の性能劣化により ~100KB が実用上限。
PDF（数百ページ、数 MB のテキスト）には対応できない。

### 将来のインデックス構成（Phase 5）

```
kukan-packages    → データセットメタデータ（変更なし）
kukan-resources   → リソースメタデータ（変更なし）
kukan-contents    → キーワード検索用テキスト（~100KB、切り詰め or チャンク分割）
kukan-embeddings  → Embedding 用小チャンク（~500 tokens × N、knn_vector フィールド付き）
```

キーワード検索と Embedding では最適なチャンクサイズが異なるため、別インデックスとする。
`kukan-contents` をチャンク分割する場合は、`collapse` フィールド（resourceId）で
リソース単位の重複排除が可能。

### 段階的な移行パス

1. **現在（Phase 4）**: 3 インデックス構成。`kukan-contents` は 1 リソース = 1 ドキュメント（100KB 切り詰め）
2. **Phase 5a**: `kukan-contents` をチャンク分割（1 リソース = N ドキュメント）。大きな PDF/CSV に対応
3. **Phase 5b**: `kukan-embeddings` を追加。AIAdapter で Embedding 生成。ハイブリッド検索（BM25 + kNN）

各ステップで変更されるのは該当インデックスのみ。`kukan-packages` / `kukan-resources` は不変。

### 決定

- 現時点では 3 インデックス分離 + 100KB 切り詰めを維持
- チャンク分割は Phase 5 で Embedding と合わせて導入
- `kukan-contents` と `kukan-embeddings` を分離する設計を前提とする

---

## 実装概要

1. Pipeline の Index ステップ（現在 no-op）を復活
2. テキスト抽出 → OpenSearch `kukan-contents` インデックスに投入
3. `kukan-packages` から nested resources を廃止、リソースメタデータは `kukan-resources` に移動
4. 検索時は `msearch` で `kukan-packages` + `kukan-resources` + `kukan-contents` を並列クエリ、アプリ層でスコアマージ
5. メタデータ再構築は `kukan-contents` に影響しない
6. `postgres.ts` は変更最小限（新メソッドを no-op で実装、既存の DB 直接クエリは変更なし）
