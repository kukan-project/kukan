# ADR-007: Data Editor をアドオンモジュールとして設計する

## ステータス

承認済み（2026-03-01）

## コンテキスト

Quality Monitorが問題を「検知」した後、データを「修正」する手段が必要。
しかし、全ての自治体がブラウザ上でのデータ編集を必要とするわけではない。
Excel等のローカルツールで編集し再アップロードするワークフローも一般的。

## 検討した選択肢

### A) コア機能として全環境必須

- 問題: 小規模自治体には過剰、バンドルサイズ増加、依存関係増

### B) 完全外部ツール（別リポジトリ）

- 問題: Quality Monitorとの連携が困難、UX分断

### C) アドオンモジュール（同一リポジトリ、独立デプロイ） — 採用

- モノレポ内に存在するが、デプロイは任意
- Quality Monitor の検知結果から直接編集画面に遷移可能
- `apps/editor` + `packages/editor-core` として分離

## 決定

Data Editor をアドオンモジュールとして設計する。
インストールは任意、Quality Monitorとの連携APIは標準提供。

## アーキテクチャ

```
apps/editor/          ← Next.js UI（独立デプロイ可能）
packages/editor-core/ ← ビジネスロジック（変更追跡、バリデーション、承認フロー）
```

### editor-core の責務

- 変更追跡（行レベルのdiff管理）
- バリデーションルールエンジン
- 承認ワークフロー（下書き → レビュー → 承認 → 公開）
- 変更履歴の永続化

### Quality Monitor との連携

```
Quality Monitor → 問題検知 → 修正提案生成 → Data Editor で修正 → 再検証
```

## 根拠

- 「カタログ」と「エディタ」は本質的に異なるユースケース
- アドオンなら段階的導入が可能（Phase 7で実装、導入は各自治体判断）
- モノレポ内なので型共有・API連携は容易
- 独立デプロイにより、エディタだけバージョンアップも可能

## 影響

- `apps/editor` はPhase 7で実装（Phase 1-6では空ディレクトリ）
- `packages/editor-core` もPhase 7
- API側に `/api/editor/*` エンドポイントをPhase 7で追加
- Phase 1-6のDB設計に `edit_session` / `edit_change` テーブルの予約枠を確保
